# Writing commit messages

## Format

All commit messages must follow this format:

```text
[TAG] Short one-line summary (max 72 characters)

Optional longer explanation of why the change is being made.
Focus on the "why" rather than the "how".
```

Key rules:

- The summary line (first line, including the tag) must be at most
  72 characters.
- Do not include any information about Claude or AI creating or
  co-authoring the commit.
- The body, if present, must be separated from the summary by a blank
  line.


## Tags

The first line must start with a tag in square brackets:

- `[#123]` — references GitHub issue #123 in this repository.
- `[#123, #456]` — references multiple issues.
- `[docs]` — documentation changes.
- `[release]` — release-related changes.
- `[infra]` — infrastructure/tooling changes.
- `[cleanup]` — small cleanup changes.

For cross-repository references, use `[Org/repo#123]`.


## Common mistakes to avoid

### AI attribution

Never add `Co-Authored-By`, "Generated by", or any other AI/Claude
attribution to commit messages. This repository explicitly forbids it.

### Summary line too long

The 72-character limit includes the tag. Count carefully.
`[#12345] Fix the handling of ...` is already 35+ characters before
the description even starts.

### Wrong tag format

Use `[#123]`, not `#123`, not `[bug]`, not `[fix]`. The tag must be
in square brackets and must be one of the listed tags or an issue
reference.

### Body describes "how" instead of "why"

Do not describe what the code change does — the diff shows that.
Explain why the change is needed.

### Fixup and squash commits

`fixup!` and `squash!` prefixes are valid commit formats. Do not
flag or "fix" them — they are intentional and used during interactive
rebases.

### Commit granularity

A commit should represent **one logical change** — the smallest
self-contained unit of work that:

- Can be described in a single, specific summary line.
- Makes sense standalone in `git log`.
- Could be reverted without undoing unrelated work.

**Too small** — merge into a larger commit or use `fixup!`:
- Reformatting or fixing a typo in code you introduced on this branch.
- Adding a missing import needed by the function you just added.
- Correcting a value or variable name introduced by the previous commit.

**Too large** — split into separate commits:
- A new feature bundled with a refactor of pre-existing code.
- Changes to multiple independent subsystems for the same issue.
- Pre-existing cleanup mixed with new work.

Each step is distinct: any one could be understood, reviewed, or reverted
independently.


### Fixup commits vs. new commits

Whenever adding a commit to an existing branch, apply this test:

> **Did this branch introduce the thing being corrected?**
>
> - **Yes** → `fixup!` the commit that introduced it.
> - **No, it existed before this branch** → new commit.

This rule applies whether the change comes from your own review, a PR
comment, or a CI failure.

**Use `fixup!`** when the change corrects something introduced by an
earlier commit on this branch — a style violation, a wrong value, a
missing import, a typo, or improved wording. The fix belongs logically
inside that original commit and will be squashed into it during rebase.

```text
fixup! [#123] The exact subject of the commit being fixed
```

The subject after the `fixup!` prefix must match the target commit's subject
exactly (git uses this for autosquash).

**Use a new commit** when the change addresses a pre-existing issue (one
that existed before this branch started) — a new feature, removing dead
code, dropping stale data, simplifying previously-correct logic,
refactoring pre-existing code. These stand on their own regardless of
whether they were prompted by a code review, a CI failure, or your own
judgement.

**One logical change per commit** — do not bundle multiple unrelated
changes into a single commit. Each logical change gets its own commit so
that the history remains bisectable and reviewable.


### Examples

**Correct `fixup!`** — on a branch documenting the fixup convention, the
second commit improved the wording of text introduced by the first:

```text
[infra] Document fixup vs. new commit convention in git-commit skill
fixup! [infra] Document fixup vs. new commit convention in git-commit skill
```

The second commit should not be `[infra] Improve wording in git-commit
skill` — the original commit on this branch is the right target.

**Correct new commits** — on the `#12345` branch, "Minor formatting fixes
in mushroom process" is a new commit (not a fixup) because the
formatting issues it fixed were pre-existing in the file, not introduced
by its branch:

```text
[#12345] Add fungal report function
[#12345] Minor formatting fixes in mushroom process   ← pre-existing
[#12345] Add fungi process manager
```

**Wrong: fixup disguised as a new commit** — if you add a function on a
branch then notice the variable name is wrong, that second change should
not be a new commit:

```text
# Wrong:
[#123] Add process_data function
[#123] Fix variable name in process_data

# Correct:
[#123] Add process_data function
fixup! [#123] Add process_data function
```
